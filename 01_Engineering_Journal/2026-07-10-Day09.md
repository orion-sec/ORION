# ORION Engineering Journal
## Day 9 – Building ORION's First Decision Engine

### Objective
Today marked the beginning of ORION's Decision Engine. The focus shifted from collecting and enriching data to allowing ORION to make simple security decisions by assigning risk scores to enriched IP addresses.

---

## What We Built

### 1. Created score.py

A new reusable module called score.py was introduced.

This module evaluates enriched IP information and assigns:

- Risk Level
- Numerical Score
- Reason for the decision

Example:

Public Global IP

↓

Medium Risk

↓

Score: 50

---

### 2. Added IP Risk Scoring

After IP enrichment, ORION now automatically evaluates every IP using score_ip().

Each enriched IP is passed into the scoring engine.

The results are stored inside:

results["IP Scores"]

---

### 3. Updated Investigation Pipeline

ORION's workflow has evolved into:

Investigation Input

↓

IOC Extraction

↓

IP Enrichment

↓

Risk Scoring

↓

SOC Report

---

### 4. Updated display.py

A new function called display_scores() was created.

The investigation report now includes:

============================
RISK SCORES
============================

Risk   : Medium
Score  : 50
Reason : Public Global IP

This gives analysts an immediate understanding of investigation priority.

---

## Engineering Challenges

While integrating the new scoring engine, display.py contained a structural issue that prevented the application from running correctly.

Instead of immediately asking for help, I decided to investigate the problem independently.

I carefully reviewed the function layout, reorganized the code structure, and tested the application repeatedly until the issue was resolved.

This was one of the biggest learning moments so far because it reinforced that debugging is a fundamental engineering skill.

---

## Lessons Learned

• Functions should have one clear responsibility.

• Data should flow through reusable modules.

• Risk decisions should be calculated separately from data extraction.

• Python error messages provide valuable debugging guidance.

• Software engineering is less about memorizing syntax and more about understanding program flow.
