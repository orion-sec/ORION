### 2. Added Analyst Recommendation Logic

Implemented a recommendation function that evaluates risk results and generates investigation guidance for the analyst.

For a Medium-risk public IP, ORION recommends:

- Enrich the IOC using threat intelligence.
- Review identity and sign-in activity.
- Validate MFA authentication status.
- Check endpoint activity in EDR.
- Search for additional users or hosts associated with the IOC.
- Escalate if suspicious activity is confirmed.

For lower-risk internal activity, ORION provides recommendations focused on validating known infrastructure, reviewing historical activity, and considering alert suppression where activity is repeatedly confirmed as expected.

Informational activity can be documented without immediate security action.

### 3. Tested Multiple IP Classification Scenarios

Three IP scenarios were tested.

#### Private IP

Test IOC:

10.0.0.5

Result:

Risk: Low
Score: 10
Reason: Private IP Address

ORION generated internal infrastructure validation recommendations.

#### Loopback IP

Test IOC:

127.0.0.1

Initial Result:

Risk: Low
Score: 10
Reason: Private IP Address

This result exposed a rule precedence issue.

The enrichment module correctly identified the IP as both private and loopback. However, the scoring engine evaluated the broader private IP condition before the more specific loopback condition.

Because the function returned immediately after matching the private condition, the loopback rule was never evaluated.

The scoring logic was rearranged so that the more specific loopback rule is evaluated before the broader private IP rule.

Final Result:

Risk: Informational
Score: 0
Reason: Loopback Address

This successfully corrected the classification.

#### Public Global IP

Test IOC:

8.8.8.8

Result:

Risk: Medium
Score: 50
Reason: Public Global IP

ORION generated a full SOC investigation recommendation workflow.

### 4. Learned Rule Precedence

A key engineering lesson from Day 10 was the importance of rule evaluation order.

The principle identified was:

Specific rules should be evaluated before broader rules.

The 127.0.0.1 test demonstrated that one value may satisfy multiple conditions.

The order of conditional statements therefore directly affects the final decision produced by the system.

This concept is relevant to:

- Detection engineering
- SIEM correlation rules
- Firewall rules
- Conditional Access policies
- SOAR playbooks
- Security decision engines

### 5. Added Investigation Priority Logic

Implemented:

determine_priority()

The function converts numerical risk scores into investigation priorities.

Current priority model:

- Score 80 or greater = P1
- Score 50 or greater = P2
- Score 20 or greater = P3
- Score below 20 = P4

The priority engine also provides an associated investigation action.

Example:

Score: 50
Priority: P2
Action: Investigate within standard SOC workflow

## Engineering Reflection

Day 10 was an important milestone in the ORION project.

ORION has moved beyond identifying indicators and now begins to interpret investigation context through structured security logic.

A significant learning point was identifying and correcting a rule precedence problem in the scoring engine.

The issue was discovered through scenario testing rather than a Python execution error. The application executed successfully but produced a logically incorrect classification.

This demonstrated the difference between code that runs successfully and code that produces the correct security decision.

The scoring and decision pipeline is beginning to reflect practical SOC investigation workflows.
