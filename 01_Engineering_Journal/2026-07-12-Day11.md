# ORION Engineering Journal – Day 11

**Date:** 12 July 2026  
**Project:** ORION Security Investigation Platform  
**Focus:** Live Threat Intelligence Integration and Threat-Aware Risk Scoring

---

## What I Built Today

Today I integrated ORION with its first live external threat intelligence API using AbuseIPDB.

I created a dedicated threat intelligence module named `threat_intel.py`.

The module performs live IP reputation lookups and returns normalised threat intelligence data including:

- IP address
- Threat intelligence source
- API availability status
- Reputation
- Confidence score
- Number of reports

I also connected the threat intelligence result to ORION's IP risk scoring engine.

ORION can now use live threat intelligence context when determining the risk level and investigation priority of a public IP address.

## New Concepts Learned

### API Integration

I learned how Python can communicate with an external threat intelligence service using HTTP requests.

The `requests` library is used to send the API request to AbuseIPDB.

### Environment Variables

I learned why API keys should not be hardcoded directly inside Python source code.

The AbuseIPDB API key is stored inside a `.env` file and loaded using `python-dotenv`.

The `.env` file is excluded from Git tracking through `.gitignore`.

### Graceful Degradation

I learned that a security investigation platform should not fail completely because one threat intelligence provider becomes unavailable.

ORION now detects when the AbuseIPDB API lookup fails and returns a normalised unavailable status.

The investigation pipeline continues using local enrichment and baseline scoring logic.

### Threat-Aware Risk Scoring

Previously, ORION classified public global IP addresses using static scoring logic.

The scoring engine can now use live threat intelligence confidence when determining risk.

Threat intelligence can therefore influence:

- Risk level
- Risk score
- Analyst recommendations
- Investigation priority

## Testing Performed

Two API scenarios were tested.

### Test 1 – Threat Intelligence Provider Unavailable

The AbuseIPDB API URL was intentionally disrupted.

ORION returned:

- Status: Unavailable
- Reputation: Unknown
- Confidence: 0
- Reports: 0

ORION continued the investigation without crashing.

The IP received baseline public IP scoring and remained P2 for further investigation.

### Test 2 – Threat Intelligence Provider Available

The AbuseIPDB API URL was restored.

ORION successfully queried AbuseIPDB.

For IP `8.8.8.8`, the API returned:

- Status: Available
- Reputation: Clean
- Confidence: 0
- Reports: 120

ORION used the live threat intelligence result and adjusted the risk decision to:

- Risk: Low
- Score: 20
- Priority: P3

This confirmed that live threat intelligence can influence the ORION decision engine.

## Mistakes and Troubleshooting

While modifying `score.py`, incorrect indentation caused multiple Python and Pylance errors.

The function body was temporarily placed outside the `score_ip()` function scope.

This caused errors including:

- Expected indented block
- Return can be used only within a function
- Undefined variables

I corrected the function indentation and removed duplicate scoring logic.

The scoring engine then executed successfully.

## SOC Engineering Lesson

Threat intelligence should enrich an investigation, not become the only source of truth.

A clean threat intelligence result does not automatically prove that activity is legitimate.

ORION should combine threat intelligence with:

- Identity activity
- Authentication history
- MFA status
- Endpoint telemetry
- Historical IOC activity
- User and host relationships

This reflects the investigation approach I use as a SOC analyst.
