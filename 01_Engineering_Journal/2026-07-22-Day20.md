# ORION Engineering Journal

## Day 20 – Introducing the ORION Processing Pipeline

### Objective

Today's objective was to begin transforming ORION from a script-based investigation engine into a platform driven by a formal processing pipeline.

The focus was architectural design rather than adding new investigation features.

---

## Work Completed

### 1. Created the ORION Processing Pipeline

Created:

11_Modules/pipeline.py

The pipeline introduces a dedicated orchestration layer responsible for executing investigation stages in sequence.

Initial implementation includes:

- OrionPipeline class
- Stage registration
- Pipeline execution
- Shared investigation state

---

### 2. Pipeline Contract

Designed the first pipeline execution contract.

Each processing stage follows the same structure:

Input:

- Investigation
- Shared Results Dictionary

Output:

- Updated Results Dictionary

This establishes a consistent interface for every future investigation module.

---

### 3. Pipeline Initialisation Stage

Created the first pipeline stage:

initialise_results_stage()

Purpose:

- Create shared investigation state
- Store the original investigation narrative
- Prepare the results dictionary for downstream processing

This stage represents the foundation of ORION's future investigation workflow.

---

### 4. Refactoring Assessment

Attempted to integrate the pipeline directly into:

07_Scripts/ioc_extractor.py

During implementation it became clear that the current investigation engine has grown significantly beyond a simple IOC extractor.

The attempted migration highlighted that introducing a processing pipeline directly into the existing script would create unnecessary complexity.

---

### 5. Architectural Review

After reviewing the current investigation engine, a strategic design decision was made.

Rather than continuing an in-place refactor, ORION will evolve using two parallel architectures.

ORION V1

- Stable Investigation Engine
- Continues to function without modification

ORION V2

- New Platform Architecture
- Pipeline-driven execution
- Gradual migration of investigation stages

---

## Architectural Decisions

Today's most significant decision was choosing not to continue refactoring the existing investigation engine.

Instead, ORION V2 will be developed alongside the stable V1 implementation.

This approach provides:

- Stable production version
- Safer migration
- Easier testing
- Clear Git history
- Easier architectural comparison
- Lower implementation risk

---

## Lessons Learned

Large architectural changes should not be forced into an existing working system.

When the architecture naturally evolves beyond its original design, introducing a new implementation alongside the existing system is often cleaner than performing an extensive in-place refactor.

This reflects a common software engineering practice for major platform redesigns.
